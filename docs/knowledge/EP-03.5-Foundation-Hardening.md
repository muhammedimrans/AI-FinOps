## Startup Lifecycle (H-001)

`AppContainer.create()` now calls `init_db()` immediately after the database engine is
created. If the database is unreachable, the exception propagates out of `create()` and the
process fails to start, rather than serving requests against an unverified database.

```
container_initializing  env=development
database_connected      pg_version="PostgreSQL 16.x"  latency_ms=12.4
container_ready         startup_ms=45.2
```

On failure:

```
database_connection_failed  error="..."  latency_ms=5000.0
[process exits — FastAPI lifespan raises the error]
```

All log events use structlog structured JSON. No stdlib `logging` calls remain in the
startup path.

## Configuration Validation (H-002)

`validate_provider_configuration(config: dict[str, Any]) -> None` inspects every key in
the JSONB `configuration` field before it is persisted to the database. Keys are normalised
(lowercased; `-` and space → `_`) before comparison against a frozenset of 23 credential
patterns:

```
api_key, apikey, secret, secret_key, secretkey, password, passwd,
access_token, accesstoken, refresh_token, refreshtoken, bearer_token,
bearertoken, auth_token, authtoken, token, client_secret, clientsecret,
private_key, privatekey, credential, credentials, authorization
```

All violations are collected and reported together in one `ValueError`, so the caller
receives the complete picture in a single round-trip. The error message explicitly names
the prohibited keys and instructs the caller to use the Secrets store.

The validator handles:
- Exact prohibited keys (`api_key`, `token`, …)
- Case variants (`API_KEY`, `ApiKey`)
- Separator variants (`api-key`, `api key`)
- Multiple simultaneous violations

## SQLAlchemy Loading Strategy (H-003)

All ORM relationships in `Organization`, `Project`, `Membership`, and `ProviderConnection`
now set `lazy="raise"`. Accessing an unloaded relationship raises
`sqlalchemy.exc.InvalidRequestError` immediately with a clear message rather than silently
emitting synchronous I/O that would crash with `MissingGreenlet` in the asyncpg driver.

Organization's three cascade relationships additionally set `passive_deletes=True`, which
tells SQLAlchemy to trust the database-level `ON DELETE CASCADE` constraints instead of
loading the full child collection into Python before deleting it.

The required loading patterns are documented in
`docs/engineering/sqlalchemy-loading-strategy.md`.

## Design Decisions

### Why `passive_deletes=True` with `lazy="raise"`?

Without `passive_deletes=True`, SQLAlchemy's ORM cascade logic loads the full child
collection before deleting the parent, which would trigger a `lazy="raise"` error. Setting
`passive_deletes=True` tells SQLAlchemy to skip that loading step and rely on the database's
`ON DELETE CASCADE` constraint instead.

### Why report all config violations at once?

A single-violation-at-a-time validator forces the caller to fix one issue, re-submit,
discover another, repeat. Collecting all violations in one pass eliminates that loop and
produces a better developer experience.

### Why `SELECT EXISTS` for `slug_exists()`?

The previous implementation fetched the full `Organization` row and discarded all columns
except the implicit existence signal. `SELECT EXISTS(SELECT 1 ...)` is explicit about intent,
avoids column hydration, and produces a constant-size result regardless of row width.

## Lessons Learned

1. **`lazy="raise"` must be paired with `passive_deletes=True` on cascade relationships.**
   Without it, any delete attempt triggers a load, which raises.
2. **Unit tests must mock `init_db`** after wiring it into `AppContainer.create()`, or they
   attempt a live database connection and fail with `ConnectionRefusedError`.
3. **Query shape changes require mock updates.** Changing `scalar_one_or_none()` to
   `scalar_one()` required updating seven mock return-value targets in `test_models_ep03.py`.
4. **`pytest.raises(match=...)` is case-sensitive by default.** Use `(?i)` prefix when the
   error message capitalisation may vary.

## Recommendations for EP-04

1. **Service layer before REST APIs.** Wire `validate_provider_configuration()` in the
   ProviderConnection service, not the API router, to keep validation close to the domain.
2. **User model should reference `AppContainer.session_factory`**, not import a session
   directly, to maintain the dependency injection pattern established in EP-03.
3. **`selectinload()` patterns are required** in every service method that returns a model
   with relationships. Document this in the PR template checklist.
4. **Secrets store integration** (TD-007) should be designed before implementing the
   ProviderConnection activation flow — the validator blocks secrets in `configuration` JSONB
   intentionally; EP-04 needs to decide where the secret reference goes.
5. **JWT issuance and Argon2 hashing** are CPU-intensive; run them in a thread pool
   (`asyncio.to_thread`) to avoid blocking the event loop.
