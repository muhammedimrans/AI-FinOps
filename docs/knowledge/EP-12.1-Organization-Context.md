# EP-12.1 — Organization Context

**Date:** 2026-06-30  
**Status:** Implemented  
**Depends on:** EP-12 Foundational Layer (auth store, DTO mappers, `VITE_ENABLE_MOCKS`)

---

## Problem Statement

Every dashboard endpoint requires `organization_id` as a query parameter. After
login, the JWT payload and `LoginResponse` contain no organization membership
data, so the frontend had no way to discover which UUID to send. A temporary
`OrgPrompt` component required manual UUID entry.

EP-12.1 closes this gap by implementing `GET /v1/organizations` and wiring the
auto-select flow into the login path.

---

## Backend Changes

### New Endpoint — `GET /v1/organizations`

**File:** `backend/app/api/v1/organizations.py`  
**Router prefix:** `/v1/organizations`  
**Auth:** `CurrentUser` (Bearer JWT required)

Returns all ACTIVE organizations the authenticated user is a member of, ordered
by membership `created_at` ascending.

**Response shape:**
```json
{
  "organizations": [
    {
      "id": "org_<hex>",
      "name": "Zero Protocol",
      "slug": "zero-protocol",
      "role": "owner"
    }
  ]
}
```

### New Schema — `backend/app/schemas/organizations.py`

- `OrgMembershipItem` — one organization with the user's role
- `OrganizationsResponse` — wrapper list

### Repository Addition — `MembershipRepository.list_by_user_email_with_orgs`

**File:** `backend/app/repositories/membership_repository.py`

New method added alongside the existing `list_by_email`. Differences:

| | `list_by_email` | `list_by_user_email_with_orgs` |
|---|---|---|
| Returns | `CursorPage[Membership]` | `list[Membership]` |
| Eager loads | Nothing (lazy="raise" would error) | `selectinload(Membership.organization)` |
| Org filter | — | ACTIVE + not soft-deleted |

### Router Registration

`backend/app/api/router.py` — added `organizations.router` with prefix `/v1`.

### Seed Script — `backend/scripts/seed_demo.py`

Idempotent script that creates demo data for development:

| Resource | Value |
|---|---|
| Organization | name="Zero Protocol", slug="zero-protocol", status=ACTIVE |
| User | email=admin@0protocol.net, password=Admin@123, status=ACTIVE, email_verified=True |
| Membership | role=OWNER |
| Project | name="AI FinOps Demo", environment=PRODUCTION |

Run with: `cd backend && python -m scripts.seed_demo`

## Frontend Changes

### `OrgSelector` component (new)

**File:** `frontend/src/components/OrgSelector.tsx`

Replaces `OrgPrompt`. On mount, calls `GET /v1/organizations` and:
- **1 org** → calls `setOrganization(id, name)` immediately; `AuthGuard` re-renders the dashboard
- **multiple orgs** → renders a picker list; user clicks to select
- **0 orgs** → shows an empty state with admin contact message
- **error** → shows inline error message

### Login flow update

**File:** `frontend/src/features/Login.tsx`

After successful `POST /v1/auth/login` + `setLogin(...)`, immediately calls
`GET /v1/organizations`. If exactly one org, calls `setOrganization(id, name)`
before navigating to `/dashboard`. This covers the common single-org case with
zero extra perceived latency (the org call is fast and sequential before
navigate).

For multi-org and zero-org users, `OrgSelector` handles it once the dashboard
route loads.

### `OrgPrompt` deleted

`frontend/src/components/OrgPrompt.tsx` has been removed. All references
updated to `OrgSelector`.

### `App.tsx`

`AuthGuard` now renders `<OrgSelector />` instead of `<OrgPrompt />` when
`organizationId` is null.

### API client — `frontend/src/lib/api.ts`

`getOrganizations()` added — calls `GET /v1/organizations`, returns
`BackendOrganizationsResponse`.

### Types — `frontend/src/types/backend.ts`

Added `BackendOrgMembershipItem` and `BackendOrganizationsResponse` mirroring
the Pydantic schemas.

## Key Design Decisions

**Why Option B (`GET /v1/organizations`) over Option A (extend `LoginResponse`)?**

The EP-12.1 spec chose Option B. It keeps `LoginResponse` unchanged (no schema
migration needed) and always reflects current membership state. The extra
round-trip is fast and sequential within the login handler.

**Why filter to ACTIVE orgs only?**

A user may retain a Membership row in a SUSPENDED or ARCHIVED organization. The
dashboard endpoints would fail or return empty data for non-ACTIVE orgs. Showing
them in the picker would confuse users, so they are excluded at the repository
layer.

**Why is the existing `list_by_email` kept?**

`list_by_email` returns a cursor-paginated `CursorPage[Membership]` without
eager-loading the Organization. It is used by admin/membership management
endpoints where Organization data is not needed. The new method is additive.

## Security Notes

- The endpoint returns only orgs the authenticated user belongs to (scoped by `current_user.email`)
- No org IDs from outside the user's memberships can be returned
- `external_id` (e.g. `org_<hex>`) is returned, not the raw UUID PK
- The seed password `Admin@123` is for development only — the seed script is in `backend/scripts/`, not deployed to production
